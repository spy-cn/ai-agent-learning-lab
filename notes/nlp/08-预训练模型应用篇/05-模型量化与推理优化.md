# 模型量化与推理优化

大语言模型虽然能力强大，但其推理成本同样惊人。一个 70B 参数的模型，仅加载就需要上百 GB 显存，生成一个 token 可能需要数百毫秒。如何让大模型跑得更快、用得更省？本章将系统讲解模型量化、KV Cache、Flash Attention、vLLM 等核心技术，带你全面掌握大模型推理优化。

## 二、模型量化

### 2.1 量化概念

量化（Quantization）是将模型从高精度（FP32/FP16）转换为低精度（INT8/INT4），以减少存储和计算开销。

```
┌─────────────────────────────────────────────────────────────┐
│                      量化原理                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  FP32 (32位浮点)：                                           │
│  符号(1) 指数(8) 尾数(23)  = 32 bits                        │
│  范围：±3.4×10³⁸    精度：~7位有效数字                       │
│  ┌────────────────────────────────────────┐                 │
│  │ 0 10000101 10100000000000000000000      │                 │
│  └────────────────────────────────────────┘                 │
│  = 41.0                                                      │
│                                                             │
│  FP16 (16位浮点)：                                           │
│  符号(1) 指数(5) 尾数(10)  = 16 bits                        │
│  范围：±65504       精度：~3位有效数字                       │
│  ┌──────────────────┐                                       │
│  │ 0 10010 0101000000 │                                      │
│  └──────────────────┘                                       │
│  = 41.0                                                      │
│                                                             │
│  INT8 (8位整数)：                                            │
│  符号(1) 数值(7)  = 8 bits                                   │
│  范围：-128 ~ 127                                            │
│  ┌────────┐                                                 │
│  │ 01001001│ = 73  → 映射到 73 × scale                      │
│  └────────┘                                                 │
│                                                             │
│  INT4 (4位整数)：                                            │
│  数值(4)  = 4 bits                                           │
│  范围：-8 ~ 7                                                │
│  ┌────┐                                                     │
│  │0100│ = 4  → 映射到 4 × scale                              │
│  └────┘                                                     │
│                                                             │
│  显存对比（7B 模型）：                                       │
│  FP32: 28 GB  →  FP16: 14 GB  →  INT8: 7 GB  →  INT4: 3.5 GB│
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 量化的数学原理

```python
# 量化的核心公式：将浮点数映射到整数范围

# === 对称量化 ===
# FP32 → INT8
# x_int = round(x_fp / scale)
# x_fp  = x_int × scale
#
# scale = max(|x|) / 127

import numpy as np

def quantize_symmetric(tensor, bits=8):
    """对称量化"""
    qmax = 2 ** (bits - 1) - 1  # INT8: 127, INT4: 7
    
    # 计算缩放因子
    abs_max = np.max(np.abs(tensor))
    scale = abs_max / qmax
    
    # 量化
    quantized = np.round(tensor / scale).clip(-qmax, qmax).astype(np.int8 if bits == 8 else np.int8)
    
    # 反量化（恢复）
    dequantized = quantized * scale
    
    return quantized, scale, dequantized

# 示例
original = np.array([0.5, -1.2, 3.4, -0.8, 2.1], dtype=np.float32)
q8, scale8, deq8 = quantize_symmetric(original, bits=8)
q4_data, scale4, deq4 = quantize_symmetric(original, bits=4)

print("FP32 原始值: ", original)
print("INT8 量化后: ", q8, f"(scale={scale8:.6f})")
print("INT8 反量化: ", deq8)
print("INT8 误差:   ", np.abs(original - deq8))
print()
print("INT4 反量化: ", deq4)
print("INT4 误差:   ", np.abs(original - deq4))
# INT4 误差明显大于 INT8
```

### 2.5 bitsandbytes 量化实战

```python
"""
使用 bitsandbytes 进行模型量化
最简单的量化方案，适合快速实验
"""
import torch
from transformers import AutoModelForCausalLM, AutoTokenizer, BitsAndBytesConfig

# === INT8 量化 ===
model_int8 = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Llama-2-7b-hf",
    load_in_8bit=True,        # 一行参数实现 INT8 量化
    device_map="auto"
)

# === INT4 量化 (NF4) ===
bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",           # NormalFloat 4
    bnb_4bit_compute_dtype=torch.float16, # 计算精度
    bnb_4bit_use_double_quant=True,       # 双重量化
)

model_int4 = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Llama-2-7b-hf",
    quantization_config=bnb_config,
    device_map="auto"
)

# === 比较显存使用 ===
def get_model_size(model):
    """获取模型显存占用（MB）"""
    total = 0
    for param in model.parameters():
        total += param.nelement() * param.element_size()
    return total / 1024 / 1024

print(f"INT8 模型大小: {get_model_size(model_int8):.0f} MB")
print(f"INT4 模型大小: {get_model_size(model_int4):.0f} MB")
```

## 三、KV Cache

### 3.1 为什么需要 KV Cache

```
┌─────────────────────────────────────────────────────────────┐
│              自回归生成的计算浪费                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  生成序列 "今天天气真好"                                     │
│                                                             │
│  Step 1: 输入 [今]                                           │
│          计算 K₁,V₁ → 输出 [天]                              │
│                                                             │
│  Step 2: 输入 [今天]                                         │
│          重新计算 K₁,V₁（浪费！）→ 再计算 K₂,V₂ → 输出 [气]   │
│                                                             │
│  Step 3: 输入 [今天气]                                       │
│          重新计算 K₁,V₁,V₂,V₂（浪费！）→ 再计算 K₃,V₃ → ...  │
│                                                             │
│  问题：之前的 K,V 每步都重新计算！                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│              KV Cache 方案                                   │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Step 1: 输入 [今]                                           │
│          计算 K₁,V₁ → 存入 Cache → 输出 [天]                 │
│                                                             │
│  ┌─────────────┐                                            │
│  │ KV Cache    │                                            │
│  │ K₁ V₁       │ ← 存下来                                   │
│  └─────────────┘                                            │
│                                                             │
│  Step 2: 输入 [天]                                           │
│          只计算 K₂,V₂ → 追加到 Cache → 输出 [气]             │
│          Q₂ × [K₁,K₂] → 使用缓存！                          │
│                                                             │
│  ┌─────────────┐                                            │
│  │ KV Cache    │                                            │
│  │ K₁ V₁       │                                            │
│  │ K₂ V₂       │ ← 追加                                     │
│  └─────────────┘                                            │
│                                                             │
│  Step 3: 输入 [气]                                           │
│          只计算 K₃,V₃ → 追加到 Cache → 输出 [真]             │
│          Q₃ × [K₁,K₂,K₃] → 使用缓存！                       │
│                                                             │
│  优化：每步只计算新 token 的 K,V，之前的从缓存读取            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 KV Cache 的代价

```python
# KV Cache 虽然加速了计算，但增加了显存占用

def kv_cache_memory(seq_len, num_layers, num_heads, head_dim, batch_size, dtype_size=2):
    """
    计算 KV Cache 的显存占用（字节）
    
    参数:
        seq_len: 序列长度
        num_layers: Transformer 层数
        num_heads: 注意力头数
        head_dim: 每个头的维度
        batch_size: 批大小
        dtype_size: 每个元素的字节数 (FP16=2)
    """
    # 每层每头需要缓存 K 和 V
    # K: [batch, heads, seq_len, head_dim]
    # V: [batch, heads, seq_len, head_dim]
    cache_size = (
        2 *                    # K 和 V
        batch_size *
        num_heads *
        seq_len *
        head_dim *
        dtype_size *
        num_layers
    )
    return cache_size

# LLaMA-7B 的 KV Cache
# num_layers=32, num_heads=32, head_dim=128
configs = [
    ("LLaMA-7B",  32, 32, 128),
    ("LLaMA-13B", 40, 40, 128),
    ("LLaMA-70B", 80, 64, 128),
]

print("KV Cache 显存占用（FP16, batch=1）：")
for name, layers, heads, dim in configs:
    for seq_len in [512, 2048, 8192]:
        mem = kv_cache_memory(seq_len, layers, heads, dim, batch_size=1)
        print(f"  {name} seq={seq_len:5d}: {mem/1024/1024/1024:.2f} GB")

# 示例输出：
# LLaMA-7B seq= 512:  0.25 GB
# LLaMA-7B seq=2048:  1.00 GB
# LLaMA-7B seq=8192:  4.00 GB  ← 序列越长，KV Cache 越大！
```

### 3.3 使用 KV Cache

```python
import torch
from transformers import AutoModelForCausalLM, AutoTokenizer

model_name = "meta-llama/Llama-2-7b-hf"
tokenizer = AutoTokenizer.from_pretrained(model_name)
model = AutoModelForCausalLM.from_pretrained(
    model_name,
    torch_dtype=torch.float16,
    device_map="auto"
)

# === 启用 KV Cache（默认就是启用的）===
input_text = "人工智能的未来"
inputs = tokenizer(input_text, return_tensors="pt").to("cuda")

# use_cache=True 启用 KV Cache
import time

# 有 KV Cache
start = time.time()
with torch.no_grad():
    outputs = model.generate(
        **inputs,
        max_new_tokens=100,
        use_cache=True    # 启用 KV Cache
    )
time_with_cache = time.time() - start
print(f"有 KV Cache: {time_with_cache:.2f}s")
print(tokenizer.decode(outputs[0]))

# 无 KV Cache（对比）
start = time.time()
with torch.no_grad():
    outputs = model.generate(
        **inputs,
        max_new_tokens=100,
        use_cache=False   # 禁用 KV Cache
    )
time_without_cache = time.time() - start
print(f"无 KV Cache: {time_without_cache:.2f}s")

print(f"\n加速比: {time_without_cache/time_with_cache:.1f}x")
```

## 四、Flash Attention

### 4.1 标准 Attention 的问题

```
┌─────────────────────────────────────────────────────────────┐
│              标准 Attention 的内存瓶颈                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Attention 计算：                                           │
│  Q [batch, heads, seq, d]                                   │
│  K [batch, heads, seq, d]                                   │
│  V [batch, heads, seq, d]                                   │
│                                                             │
│  步骤1: S = Q × K^T   [batch, heads, seq, seq]              │
│         ↑ seq×seq 的注意力分数矩阵！                         │
│                                                             │
│  步骤2: P = softmax(S) [batch, heads, seq, seq]             │
│         ↑ 另一个 seq×seq 矩阵                               │
│                                                             │
│  步骤3: O = P × V     [batch, heads, seq, d]                │
│                                                             │
│  问题：                                                     │
│  seq=4096 → S 矩阵大小 = 4096×4096×2bytes = 32MB (每头每批)  │
│  seq=32768 → S 矩阵大小 = 32768×32768×2bytes = 2GB！        │
│                                                             │
│  这些中间矩阵导致：                                          │
│  1. 显存占用大                                              │
│  2. HBM 读写次数多（GPU显存带宽是瓶颈）                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 Flash Attention 原理

```
┌─────────────────────────────────────────────────────────────┐
│                  Flash Attention 原理                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  核心思想：分块计算（Tiling），不实例化完整 S 矩阵            │
│                                                             │
│  标准 Attention：                                           │
│  ┌─────────────────────────────────────┐                    │
│  │  Q  │   完整的 seq×seq 矩阵 S       │                    │
│  │  K  │   全部写入 HBM（显存）         │                    │
│  │  V  │   再读取做 softmax            │                    │
│  └─────────────────────────────────────┘                    │
│  HBM 读写量：O(seq² × d)                                   │
│                                                             │
│  Flash Attention：                                          │
│  ┌─────────────────────────────────────┐                    │
│  │  将 Q, K, V 分成小块                 │                    │
│  │                                     │                    │
│  │  Q₁ K₁ V₁  → 在 SRAM 算出部分结果   │                    │
│  │  Q₁ K₂ V₂  → 增量更新              │                    │
│  │  Q₁ K₃ V₃  → 增量更新              │                    │
│  │  ...                                │                    │
│  │  Q₂ K₁ V₁  → 在 SRAM 算出部分结果   │                    │
│  │  ...                                │                    │
│  │                                     │                    │
│  │  关键：softmax 可以增量计算          │                    │
│  │  （Online Softmax / 分块 Softmax）   │                    │
│  └─────────────────────────────────────┘                    │
│  HBM 读写量：O(seq × d²)  ← 少很多！                        │
│                                                             │
│  GPU 内存层次：                                             │
│  ┌──────────┐  大但慢                                      │
│  │   HBM    │  ~80 GB, ~2 TB/s                            │
│  │  (显存)  │                                              │
│  ├──────────┤                                              │
│  │  SRAM    │  小但快                                      │
│  │ (共享内存)│ ~164 KB/block, ~19 TB/s                     │
│  ├──────────┤                                              │
│  │ Registers│ 最快                                         │
│  └──────────┘                                              │
│                                                             │
│  Flash Attention 尽量在 SRAM 中完成计算                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

```python
def benchmark_attention(model, input_ids, num_runs=10):

        torch.cuda.synchronize()
        start = time.time()
        with torch.no_grad():
            model.generate(**input_ids, max_new_tokens=50, use_cache=True)

        times.append(time.time() - start)
```

## 五、vLLM 推理框架

### 5.1 vLLM 与 PagedAttention

```
┌─────────────────────────────────────────────────────────────┐
│                  PagedAttention 原理                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  传统 KV Cache 存储：                                       │
│  ┌─────────────────────────────────────────────┐            │
│  │  Request 1: [K₁V₁ K₂V₂ K₃V₃ _ _ _ _ _ _ ]  │ 连续存储   │
│  │  Request 2: [K₁V₁ K₂V₂ _ _ _ _ _ _ _ _ _ ]  │ 预分配     │
│  │  Request 3: [K₁V₁ K₂V₂ K₃V₃ K₄V₄ K₅V₅ _ ] │ 最大长度   │
│  │                                              │            │
│  │  问题：                                       │            │
│  │  1. 预分配最大长度 → 大量浪费                 │            │
│  │  2. 内存碎片化                                │            │
│  │  3. 无法共享（如 beam search）                │            │
│  └─────────────────────────────────────────────┘            │
│                                                             │
│  PagedAttention（类似操作系统的虚拟内存）：                   │
│  ┌─────────────────────────────────────────────┐            │
│  │  物理内存按固定大小的 Block 划分               │            │
│  │                                              │            │
│  │  Block 0: [K₁V₁ K₂V₂]  ← Req 1              │            │
│  │  Block 1: [K₁V₁ K₂V₂]  ← Req 2              │            │
│  │  Block 2: [K₃V₃ K₄V₄]  ← Req 1              │            │
│  │  Block 3: [K₃V₃ _    ]  ← Req 2              │            │
│  │  Block 4: [K₅V₅ _    ]  ← Req 1              │            │
│  │                                              │            │
│  │  逻辑视图 → 物理块映射（页表）                 │            │
│  │                                              │            │
│  │  Req 1: Block0 → Block2 → Block4            │            │
│  │  Req 2: Block1 → Block3                     │            │
│  │                                              │            │
│  │  优点：                                       │            │
│  │  1. 按需分配，无浪费                          │            │
│  │  2. 无内存碎片                                │            │
│  │  3. 支持共享（Prefix Sharing）                │            │
│  └─────────────────────────────────────────────┘            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 6.1 为什么需要模型并行

```
┌─────────────────────────────────────────────────────────────┐
│                  单卡装不下怎么办                             │
├─────────────────────────────────────────────────────────────┤

```

### 6.2 并行策略对比

```
┌─────────────────────────────────────────────────────────────┐
│                三种模型并行策略                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 张量并行（Tensor Parallel）                              │
│  ┌─────────────────────────────────────────────┐            │
│  │  将权重矩阵按维度切分到不同 GPU                │            │
│  │                                              │            │
│  │  GPU 0: W[:, :d/2]    GPU 1: W[:, d/2:]      │            │
│  │                                              │            │
│  │  Y₀ = X × W₀          Y₁ = X × W₁            │            │
│  │     │                    │                    │            │
│  │     └──── AllReduce ──────┘                   │            │
│  │              Y = Y₀ + Y₁                     │            │
│  │                                              │            │
│  │  特点：每层都拆分，通信频繁                    │            │
│  │  适合：层内参数大的模型                        │            │
│  └─────────────────────────────────────────────┘            │
│                                                             │
│  2. 流水线并行（Pipeline Parallel）                          │
│  ┌─────────────────────────────────────────────┐            │
│  │  将不同层分配到不同 GPU                       │            │
│  │                                              │            │
│  │  GPU 0         GPU 1         GPU 2           │            │
│  │  ┌────┐       ┌────┐       ┌────┐           │            │
│  │  │ L0 │──→    │ L1 │──→    │ L2 │──→ Output │            │
│  │  │ L1 │ batch │ L2 │ batch │ L3 │           │            │
│  │  │ L2 │  0    │ L3 │  0    │ L4 │           │            │
│  │  └────┘       └────┘       └────┘           │            │
│  │                                              │            │
│  │  特点：通信少但有流水线气泡                    │            │
│  │  适合：层数多的模型                            │            │
│  └─────────────────────────────────────────────┘            │
│                                                             │
│  3. 数据并行（Data Parallel）                                │
│  ┌─────────────────────────────────────────────┐            │
│  │  每张 GPU 放完整模型副本                      │            │
│  │  处理不同的数据                                │            │
│  │                                              │            │
│  │  GPU 0 (完整模型) → batch 0                   │            │
│  │  GPU 1 (完整模型) → batch 1                   │            │
│  │                                              │            │
│  │  特点：不省显存但提高吞吐量                    │            │
│  └─────────────────────────────────────────────┘            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

| 并行策略 | 拆分对象 | 通信量 | 显存节省 | 实现难度 |
|---------|---------|--------|---------|---------|
| 数据并行 | 数据 | 中（梯度同步）| 不节省 | 低 |
| 张量并行 | 层内权重 | 高（每层同步）| 线性节省 | 高 |
| 流水线并行 | 层间分配 | 低（层间传递）| 线性节省 | 中 |
| 3D 并行 | 组合以上 | 最优 | 最大 | 最高 |

### 8.4 性能基准测试

```python
"""
推理性能基准测试模板
"""
import time
import torch
from transformers import AutoModelForCausalLM, AutoTokenizer

def benchmark_model(
    model_name,
    quantization=None,
    batch_sizes=[1, 4, 8, 16],
    seq_lengths=[128, 512, 1024],
    gen_length=100,
):
    """基准测试函数"""
    tokenizer = AutoTokenizer.from_pretrained(model_name)
    
    # 加载模型
    kwargs = {"torch_dtype": torch.float16, "device_map": "auto"}
    if quantization == "int8":
        kwargs["load_in_8bit"] = True
    elif quantization == "int4":
        from transformers import BitsAndBytesConfig
        kwargs["quantization_config"] = BitsAndBytesConfig(
            load_in_4bit=True,
            bnb_4bit_compute_dtype=torch.float16
        )
    
    model = AutoModelForCausalLM.from_pretrained(model_name, **kwargs)
    
    results = []
    for batch_size in batch_sizes:
        for seq_len in seq_lengths:
            # 准备输入
            input_ids = torch.randint(
                0, tokenizer.vocab_size, (batch_size, seq_len)
            ).to(model.device)
            
            # 预热
            model.generate(input_ids, max_new_tokens=5, use_cache=True)
            torch.cuda.synchronize()
            
            # 计时
            start = time.time()
            model.generate(
                input_ids, max_new_tokens=gen_length, use_cache=True
            )
            torch.cuda.synchronize()
            elapsed = time.time() - start
            
            # 计算指标
            tokens_generated = batch_size * gen_length
            throughput = tokens_generated / elapsed
            
            results.append({
                "batch_size": batch_size,
                "seq_len": seq_len,
```

## 小结

| 要点 | 内容 |
|------|------|
| 推理挑战 | 显存墙（参数大）、计算墙（算量大）、带宽墙（读取慢）|
| 模型量化 | FP16→INT8→INT4，减少显存和带宽需求 |
| 量化方案 | bitsandbytes（简单）、GPTQ（精确）、AWQ（快速）|
| KV Cache | 缓存历史 K,V 避免重复计算，自回归生成必备 |
| Flash Attention | 分块计算避免大矩阵，减少 HBM 读写 |
| vLLM | PagedAttention + Continuous Batching，高吞吐推理 |
| 模型并行 | 张量并行（拆权重）、流水线并行（拆层）|
| 框架选择 | vLLM（通用推荐）、TensorRT-LLM（极致性能）、llama.cpp（CPU）|
| 部署 | 量化 + KV Cache + vLLM + 负载均衡 |
