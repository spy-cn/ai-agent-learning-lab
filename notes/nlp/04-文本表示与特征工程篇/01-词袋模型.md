# 词袋模型（Bag of Words）

词袋模型（Bag of Words, BOW）是 NLP 中最经典、最基础的文本表示方法。它的核心思想极其简洁：**把文本看作一袋无序的词，仅通过词的出现频次来表示文档**。尽管简单，词袋模型却是理解 TF-IDF、主题模型乃至词嵌入的重要起点，至今仍广泛用于文本分类、信息检索等任务中。

---

## 一、核心思想

词袋模型的关键假设：

- **忽略词序**：只关心"哪些词出现了"，不关心"以什么顺序出现"
- **忽略语法**：没有主谓宾结构的概念
- **只统计词频**：用每个词的出现次数作为特征值

一句话概括：**"我爱NLP" 和 "NLP爱我" 在词袋模型下是完全相同的表示**。

> **直观类比**：把文档想象成一个装满词的袋子，伸手进去摸一把，你只知道袋子里有哪些词、各有多少个，却不知道它们原本是怎么排列的。

---

## 二、构建步骤

词袋模型的构建可分为四个步骤：

```
┌─────────────────────────────────────────────────────────┐
│           词袋模型构建流程                                │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  文档集合                                                │
│  ┌──────────────────────┐                               │
│  │ D1: 我 爱 自然 语言   │                               │
│  │ D2: 自然 语言 很 有趣 │                               │
│  │ D3: 我 爱 编程        │                               │
│  └──────────┬───────────┘                               │
│             │                                           │
│             ▼                                           │
│  ① 分词（Tokenization）                                 │
│  ┌──────────────────────┐                               │
│  │ D1: [我, 爱, 自然, 语言]                              │
│  │ D2: [自然, 语言, 很, 有趣]                            │
│  │ D3: [我, 爱, 编程]                                   │
│  └──────────┬───────────┘                               │
│             │                                           │
│             ▼                                           │
│  ② 构建词表（Vocabulary）                               │
│  ┌──────────────────────┐                               │
│  │ 词表: {我, 爱, 自然, 语言, 很, 有趣, 编程}            │
│  │ 索引:   0   1   2     3    4    5    6               │
│  └──────────┬───────────┘                               │
│             │                                           │
│             ▼                                           │
│  ③ 统计词频（Count）                                    │
│  ┌──────────────────────┐                               │
│  │ D1: {我:1, 爱:1, 自然:1, 语言:1}                     │
│  │ D2: {自然:1, 语言:1, 很:1, 有趣:1}                   │
│  │ D3: {我:1, 爱:1, 编程:1}                             │
│  └──────────┬───────────┘                               │
│             │                                           │
│             ▼                                           │
│  ④ 生成向量（Vector）                                   │
│  ┌─────────────────────────────────────────┐            │
│  │     我  爱  自然 语言 很  有趣 编程      │            │
│  │ D1 [1,  1,  1,   1,   0,  0,   0  ]     │            │
│  │ D2 [0,  0,  1,   1,   1,  1,   0  ]     │            │
│  │ D3 [1,  1,  0,   0,   0,  0,   1  ]     │            │
│  └─────────────────────────────────────────┘            │
│             │                                           │
│             ▼                                           │
│      文档-词矩阵（Document-Term Matrix）                 │
└─────────────────────────────────────────────────────────┘
```

最终得到的矩阵称为**文档-词矩阵**（Document-Term Matrix, DTM），或其转置**词-文档矩阵**（TDM）。

### 文档-词矩阵示意

```
         word_0  word_1  word_2  word_3  ...  word_V
Doc_0  [   2       0       1       0     ...    0   ]
Doc_1  [   0       1       3       0     ...    2   ]
Doc_2  [   1       0       0       1     ...    0   ]
  ⋮        ⋮       ⋮       ⋮       ⋮              ⋮
Doc_N  [   0       2       0       1     ...    1   ]

其中 V = 词表大小, N = 文档数
矩阵中每个元素 = 词 j 在文档 i 中出现的次数
```

## 三、n-gram 模型

词袋模型最大的缺陷是丢失了词序信息。**n-gram**（n元语法）是一种折中方案：把相邻的 n 个词作为一个"词组单元"。

### n-gram 分类

| 类型 | n 值 | 示例（句子："自然 语言 处理 很 有趣"） |
|------|------|----------------------------------------|
| unigram | 1 | 自然, 语言, 处理, 很, 有趣 |
| bigram | 2 | 自然语言, 语言处理, 处理很, 很有趣 |
| trigram | 3 | 自然语言处理, 语言处理很, 处理很有趣 |

### 为什么需要 n-gram？

```
单用 unigram 的问题：
┌──────────────────────────────────────────┐
│ 句子A: "这部电影  好   看   不  坏"        │
│ 句子B: "这部电影  好   看   吗  ？不算坏"  │
│                                          │
│ unigram 下两者都含"好""看"，难以区分     │
│                                          │
│ bigram 能捕捉：                           │
│   A 含 "好看"（正面）                      │
│   B 含 "好看吗"（疑问）                    │
└──────────────────────────────────────────┘
```

### n-gram 示例代码

```python
from sklearn.feature_extraction.text import CountVectorizer
import jieba

# Sample corpus
corpus = [
    "自然语言处理是人工智能的重要方向",
    "自然语言处理很有趣",
    "深度学习改变了自然语言处理"
]

# Chinese tokenization: jieba cuts text into words, then join with space
def tokenize_zh(text):
    return " ".join(jieba.cut(text))

tokenized = [tokenize_zh(doc) for doc in corpus]

# --- Unigram only ---
vectorizer_uni = CountVectorizer(ngram_range=(1, 1))
X_uni = vectorizer_uni.fit_transform(tokenized)
print("Unigram feature names:")
print(vectorizer_uni.get_feature_names_out())
# Output: ['人工智能' '处理' '方向' '很有趣' '是' '深度' '自然' '自然语言' '语言' '改变' '重要' '学习']

# --- Unigram + Bigram ---
vectorizer_bi = CountVectorizer(ngram_range=(1, 2))
X_bi = vectorizer_bi.fit_transform(tokenized)
print("\nUnigram + Bigram feature names:")
print(vectorizer_bi.get_feature_names_out())
# Includes both single words and word pairs like '自然语言 处理'

# --- Shape comparison ---
print(f"\nUnigram matrix shape: {X_uni.shape}")   # (3, 12)
print(f"Bigram  matrix shape: {X_bi.shape}")      # (3, 25+)
```

### n-gram 选择建议

| n 值 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 1 (unigram) | 特征少、计算快 | 无词序、无语义 | 基础分类、主题分析 |
| 2 (bigram) | 捕捉局部词序 | 特征膨胀 | 情感分析、短语识别 |
| 3 (trigram) | 更丰富的上下文 | 特征爆炸、稀疏 | 特定领域、命名实体 |
| 4+ | 极少使用 | 过度稀疏 | 几乎不用 |

> **实践建议**：通常使用 `ngram_range=(1, 2)` 即可兼顾效果和效率。

## 四、Hash Trick（特征哈希）

当词表极大（百万级以上）时，构建和维护词表会消耗大量内存。**Hash Trick**（哈希技巧）通过哈希函数将词直接映射到固定大小的向量空间，**无需构建词表**。

### Hash Trick 原理

```
┌──────────────────────────────────────────────────┐
│              Hash Trick 工作原理                  │
├──────────────────────────────────────────────────┤
│                                                  │
│  传统方式：                                      │
│    词 → 查词表 → 得到索引 → 更新向量              │
│    需要存储: {词: 索引} 字典                      │
│                                                  │
│  Hash Trick：                                    │
│    词 → hash(词) % n_features → 直接得到索引      │
│    无需存储词表！                                │
│                                                  │
│  示例（n_features = 10）：                        │
│                                                  │
│    "自然" → hash("自然") % 10 = 3  → vec[3] += 1 │
│    "语言" → hash("语言") % 10 = 7  → vec[7] += 1 │
│    "处理" → hash("处理") % 10 = 3  → vec[3] += 1 │
│                                                  │
│  注意："自然"和"处理"哈希到了同一个位置（冲突）   │
└──────────────────────────────────────────────────┘
```

### Hash Trick 示例代码

```python
from sklearn.feature_extraction.text import HashingVectorizer

# HashingVectorizer: no vocabulary storage needed
hash_vectorizer = HashingVectorizer(
    n_features=2**10,      # Hash space size (1024)
    alternate_sign=False,  # Use positive counts only
    ngram_range=(1, 2)     # Include bigrams
)

tokenized = [" ".join(jieba.cut(doc)) for doc in corpus]
X_hash = hash_vectorizer.transform(tokenized)

print(f"Hash matrix shape: {X_hash.shape}")   # (3, 1024)
print(f"Non-zero entries:  {X_hash.nnz}")      # Number of non-zero elements
# Output:
#   Hash matrix shape: (3, 1024)
#   Non-zero entries:  25
```

### Hash Trick 优缺点

| 优点 | 缺点 |
|------|------|
| 无需存储词表，节省内存 | 存在哈希冲突 |
| 在线学习，可流式处理 | 无法反查特征对应的词 |
| 适合超大规模数据 | 冲突会降低模型性能 |
| `space = O(n_features)` 固定 | 解释性差 |

> **冲突缓解**：增大 `n_features`（如 2^24）；或使用带符号哈希（`alternate_sign=True`）让冲突在期望上相互抵消。

## 五、CountVectorizer 完整用法

`CountVectorizer` 是 scikit-learn 提供的词袋模型实现，功能完善且易于使用。

### 完整示例

```python
from sklearn.feature_extraction.text import CountVectorizer
import jieba
import pandas as pd

# ---------- 1. Prepare corpus ----------
corpus = [
    "自然语言处理是人工智能的重要分支",
    "机器学习和深度学习是AI的核心技术",
    "自然语言处理与计算机视觉是两大方向",
    "深度学习推动了自然语言处理的突破"
]

# Chinese text needs explicit tokenization
def chinese_tokenizer(text):
    return list(jieba.cut(text))

# ---------- 2. Configure CountVectorizer ----------
vectorizer = CountVectorizer(
    tokenizer=chinese_tokenizer,   # Custom Chinese tokenizer
    token_pattern=None,            # Disable default pattern when using custom tokenizer
    lowercase=True,                # Lowercase (default True)
    stop_words=None,               # No stopword removal here
    ngram_range=(1, 2),            # Unigrams + bigrams
    max_features=50,               # Keep only top-50 features
    min_df=1,                      # Min document frequency
    max_df=0.95,                   # Max document frequency (ignore overly common terms)
    binary=False                   # If True, use 0/1 instead of counts
)

# ---------- 3. Fit and transform ----------
X = vectorizer.fit_transform(corpus)

# ---------- 4. Inspect results ----------
print("词表大小:", len(vectorizer.vocabulary_))
print("矩阵形状:", X.shape)
print("\n前20个特征词:")
print(vectorizer.get_feature_names_out()[:20])

# Convert to dense DataFrame for visualization (small data only)
df = pd.DataFrame(
    X.toarray(),
    columns=vectorizer.get_feature_names_out(),
    index=[f"Doc{i+1}" for i in range(len(corpus))]
)
print("\n文档-词矩阵（前5列）:")
print(df.iloc[:, :8])

# ---------- 5. Transform new documents ----------
new_docs = ["自然语言处理很有趣"]
X_new = vectorizer.transform(new_docs)
print(f"\n新文档向量非零元素数: {X_new.nnz}")

# ---------- 6. Access vocabulary mapping ----------
print("\n词汇映射示例:")
for word, idx in list(vectorizer.vocabulary_.items())[:5]:
    print(f"  '{word}' → index {idx}")
```

### CountVectorizer 关键参数详解

| 参数 | 类型 | 说明 | 常用值 |
|------|------|------|--------|
| `tokenizer` | callable | 自定义分词函数 | `jieba.cut` |
| `ngram_range` | tuple | n-gram 范围 | `(1,1)`, `(1,2)` |
| `max_features` | int | 保留最大特征数 | 根据数据规模 |
| `min_df` | int/float | 最小文档频率 | 1-5 或 0.01-0.05 |
| `max_df` | float | 最大文档频率 | 0.90-0.95 |
| `stop_words` | list | 停用词列表 | 自定义或 'english' |
| `binary` | bool | 是否二值化 | False（默认） |
| `lowercase` | bool | 是否小写化 | True |

## 六、词袋模型的局限性

```
┌───────────────────────────────────────────────────────────────┐
│                   词袋模型的三大局限                           │
├───────────────────────────────────────────────────────────────┤
│                                                               │
│  ① 稀疏性（Sparsity）                                         │
│     ┌────────────────────────────────────┐                    │
│     │ 词表 V=100,000，文档含100词         │                    │
│     │ 非零比例 = 100/100000 = 0.1%        │                    │
│     │ 99.9% 的向量元素是 0！              │                    │
│     │ → 内存浪费、计算低效、维度灾难       │                    │
│     └────────────────────────────────────┘                    │
│                                                               │
│  ② 无语义（No Semantics）                                     │
│     ┌────────────────────────────────────┐                    │
│     │ "狗" 和 "犬" 完全不同的特征          │                    │
│     │ "开心" 和 "快乐" 没有任何关联        │                    │
│     │ 无法理解词与词之间的相似度           │                    │
│     └────────────────────────────────────┘                    │
│                                                               │
│  ③ 无词序（No Word Order）                                    │
│     ┌────────────────────────────────────┐                    │
│     │ "狗咬人" = "人咬狗"                 │                    │
│     │ "我打败了你" = "你打败了我"         │                    │
│     │ 语义完全相反但表示完全相同           │                    │
│     └────────────────────────────────────┘                    │
│                                                               │
│  这些局限催生了后续：                                          │
│    • TF-IDF → 缓解稀疏性和频率偏向            │                │
│    • Word2Vec/GloVe → 引入语义信息             │               │
│    • RNN/Transformer → 捕捉词序和上下文        │               │
└───────────────────────────────────────────────────────────────┘
```

### 词袋模型 vs 后续方法

| 维度 | 词袋模型 | TF-IDF | Word2Vec | BERT |
|------|---------|--------|----------|------|
| 词序信息 | 无 | 无 | 局部 | 完整 |
| 语义信息 | 无 | 无 | 有 | 丰富 |
| 稀疏性 | 高 | 高 | 低 | 低 |
| 上下文感知 | 否 | 否 | 否 | 是 |
| 计算成本 | 低 | 低 | 中 | 高 |

## 七、综合示例：文本分类

```python
from sklearn.feature_extraction.text import CountVectorizer, TfidfTransformer
from sklearn.naive_bayes import MultinomialNB
from sklearn.pipeline import Pipeline
from sklearn.model_selection import train_test_split
from sklearn.metrics import classification_report
import jieba

# ---------- Toy dataset: sentiment classification ----------
texts = [
    "这部电影真的太好看了 剧情精彩 演技在线",
    "太烂了 浪费时间 剧情无聊 演技差",
    "非常好看 强烈推荐 值得二刷",
    "无聊透顶 什么烂片 浪费钱",
    "剧情紧凑 演员表现出色 好片",
    "难看 故事老套 不推荐",
    "经典之作 百看不厌 演技巅峰",
    "失望 剧情拖沓 没意思"
] * 10  # Repeat to have more samples

labels = [1, 0, 1, 0, 1, 0, 1, 0] * 10  # 1=positive, 0=negative

# ---------- Build pipeline: CountVectorizer + Naive Bayes ----------
def chinese_tokenizer(text):
    return list(jieba.cut(text))

pipe = Pipeline([
    ('vect', CountVectorizer(tokenizer=chinese_tokenizer, token_pattern=None,
                             ngram_range=(1, 2), min_df=2)),
    ('clf', MultinomialNB())
])

# ---------- Train-test split and evaluation ----------
X_train, X_test, y_train, y_test = train_test_split(
    texts, labels, test_size=0.3, random_state=42
)

pipe.fit(X_train, y_train)
y_pred = pipe.predict(X_test)

print("=== 词袋模型 + 朴素贝叶斯 文本分类报告 ===")
print(classification_report(y_test, y_pred, target_names=["负面", "正面"]))

# ---------- Predict new text ----------
new_text = ["这部电影真的很好看 推荐给大家"]
pred = pipe.predict(new_text)
print(f"预测结果: {'正面' if pred[0]==1 else '负面'}")
```

## 小结

| 要点 | 内容 |
|------|------|
| 核心思想 | 忽略词序和语法，只统计词频，将文本转为定长向量 |
| 构建步骤 | 分词 → 构建词表 → 统计词频 → 生成文档-词矩阵 |
| n-gram | 把相邻 n 个词作为单元，部分恢复词序信息 |
| Hash Trick | 用哈希函数代替词表，适合超大规模数据，但有冲突 |
| 工具 | sklearn 的 `CountVectorizer` 和 `HashingVectorizer` |
| 主要优点 | 实现简单、可解释性强、计算高效 |
| 主要缺点 | 稀疏性高、无语义、无词序、维度灾难 |
| 适用场景 | 文本分类、信息检索基线、特征工程的基础环节 |
| 后续演进 | TF-IDF（加权）、Word2Vec（语义）、BERT（上下文） |
