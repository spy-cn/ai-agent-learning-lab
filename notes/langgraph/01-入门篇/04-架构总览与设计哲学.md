## 核心抽象

### 1. StateGraph（状态图）

图是一等公民。StateGraph 是整个应用的**骨架**：

```python
graph = StateGraph(State)
graph.add_node("node_a", function_a)
graph.add_node("node_b", function_b)
graph.add_edge(START, "node_a")
graph.add_edge("node_a", "node_b")
graph.add_edge("node_b", END)

compiled = graph.compile()
```

### StateGraph 生命周期

```
创建 ──────────────── 编译 ──────────────── 执行
  │                    │                    │
  │ add_node()         │ compile()          │ invoke()
  │ add_edge()         │  生成可执行图       │ stream()
  │ add_conditional_   │  接入 checkpointer  │
  │   edges()          │  设置中断点         │
  │                    │                    │
  ▼                    ▼                    ▼
[构建拓扑结构]      [生成Pregel执行器]    [运行图并追踪]
```

---

### 2. State（状态）

State 是贯穿整个图的**数据载体**，用 TypedDict 或 Pydantic 定义。

#### 状态设计模式

```python
# 模式一：简单对话
class ChatState(TypedDict):
    messages: Annotated[list, add_messages]

# 模式二：RAG 应用
class RAGState(TypedDict):
    messages: Annotated[list, add_messages]
    question: str
    retrieved_docs: list
    answer: str

# 模式三：多 Agent 系统
class MultiAgentState(TypedDict):
    messages: Annotated[list, add_messages]
    current_agent: str
    task_list: list
    completed_tasks: Annotated[list, operator.add]  # 追加合并
    pending_review: bool
```

#### 状态流转图

```
          ┌───────────────────────────────────┐
          │             State                 │
          │  ┌─────────┐  ┌──────────┐       │
          │  │messages │  │question  │       │
          │  └─────────┘  └──────────┘       │
          │  ┌─────────┐  ┌──────────┐       │
          │  │  docs   │  │  answer  │       │
          │  └─────────┘  └──────────┘       │
          └───────────────────────────────────┘
                          │
              ┌───────────┼───────────┐
              │           │           │
              ▼           ▼           ▼
          ┌────────┐ ┌────────┐ ┌────────┐
          │ Node A │ │ Node B │ │ Node C │
          │读取+修改│ │读取+修改│ │读取+修改│
          └────────┘ └────────┘ └────────┘
              │           │           │
              └───────────┼───────────┘
                          │
                          ▼
                    合并更新到 State
                  （通过 Reducer 策略）
```

---

### 3. Node（节点）

节点是执行具体逻辑的函数。所有节点遵循统一接口：

```python
# 标准节点函数
def my_node(state: State) -> dict:
    """
    输入: 完整状态
    输出: 状态更新（部分字段）
    """
    pass

# 节点也可以是 Runnable（如 LLM Chain）
graph.add_node("llm", my_llm_chain)  # Chain/LLM 对象也可以作为节点
```

#### 节点类型

| 类型 | 说明 | 示例 |
|------|------|------|
| **LLM 节点** | 调用大模型 | `llm.invoke(state)` |
| **工具节点** | 执行外部工具 | 数据库查询、API 调用 |
| **处理节点** | 数据转换 | 格式化、过滤、解析 |
| **路由节点** | 决策分支 | 根据条件选择路径 |
| **IO 节点** | 人机交互 | 等待用户输入、输出结果 |
| **子图节点** | 嵌套图 | 另一个编译后的图 |

---

### 4. Edge（边）

边定义节点间的控制流。LangGraph 提供三种边：

```python
# 1. 普通边：固定连接
graph.add_edge("node_a", "node_b")

# 2. 条件边：动态路由
graph.add_conditional_edges(
    "node_a",               # 源节点
    routing_function,        # 路由函数，返回目标节点名
    {
        "path_1": "node_b",
        "path_2": "node_c",
        "end": END
    }
)

# 3. 入口/出口边
graph.add_edge(START, "first_node")
graph.add_edge("last_node", END)
```

#### 边的类型图解

```
普通边（无条件）:
    [Node A] ──────► [Node B]

条件边（有分支）:
                    ┌──► [Node B]  (if condition_1)
    [Node A] ───────┼──► [Node C]  (if condition_2)
                    └──► [  END  ]  (if done)

循环边（反馈环）:
    [Node A] ──► [Node B] ──► [检查]
                                    │
                    ┌───────────────┘
                    │ (条件不满足)
                    ▼
                回到 [Node A]
```

### 5. Reducer（状态合并策略）

Reducer 是 LangGraph 的精髓之一——它定义了多个节点的输出如何合并。

```python
import operator
from typing import Annotated
from typing_extensions import TypedDict

class State(TypedDict):
    # 追加模式：新列表追加到旧列表
    messages: Annotated[list, add_messages]

    # 累加模式：使用 operator.add
    collected_items: Annotated[list, operator.add]

    # 覆盖模式：默认（不标注 Annotated）
    current_answer: str

    # 自定义 reducer
    counter: Annotated[int, lambda old, new: old + new]
```

#### Reducer 工作原理

```
当前 State:
  messages: [msg1, msg2]

节点返回:
  {"messages": [msg3]}

        │ apply_messages_reducer
        ▼

更新后 State:
  messages: [msg1, msg2, msg3]   ← 追加，不是覆盖
```

## 执行模型：Pregel

LangGraph 底层使用 **Pregel 模型**（灵感来自 Google Pregel 图计算框架）来执行图：

```
Pregel 执行 = 多轮 Superstep（超步）

Superstep 1:  [START] → 执行入口节点
Superstep 2:  执行第一批节点，合并状态
Superstep 3:  执行第二批节点，合并状态
    ...
Superstep N:  到达 END，返回结果
```

### 执行示例

```
图结构:
  START → A → B → C → END
              ↓
              D → END

Superstep 1:  执行 A
Superstep 2:  并行执行 B 和 D（如果它们在同一层）
Superstep 3:  执行 C
Superstep 4:  到达 END
```

### recursion_limit

为防止无限循环，默认最多 25 步：

```python
# 调整最大步数
result = graph.invoke(
    input,
    config={"recursion_limit": 50}
)
```

## 设计哲学

### 1. 低抽象（Low Abstraction）

LangGraph 有意**保持低抽象层次**——你直接操作图、节点、边，而不是被隐藏在魔法装饰器后面。

```
高抽象框架:     @agent   →  "自动"完成一切（黑盒）
                         ↓
LangGraph:     graph.add_node("reason", reason_fn)
               graph.add_edge("reason", "act")
               你完全知道每一步在做什么（白盒）
```

### 2. 显式优于隐式（Explicit > Implicit）

- 状态是显式的（TypedDict 定义）
- 控制流是显式的（add_edge 代码可见）
- 工具绑定是显式的（bind_tools）
- 中断点是显式的（interrupt_before/after）

### 3. 可组合性（Composability）

一切都可以组合：

```python
# 子图组合
rag_graph = build_rag().compile()
code_graph = build_code_agent().compile()
main_graph = StateGraph(State)
main_graph.add_node("rag", rag_graph)      # 子图作为节点
main_graph.add_node("code", code_graph)

# 与 LangChain 组件组合
from langchain_core.prompts import ChatPromptTemplate
chain = ChatPromptTemplate.from_template("...") | llm | parser
graph.add_node("chain", chain)  # Chain 作为节点
```

### 4. 可观测性（Observability）

```python
# 每一步都自动追踪到 LangSmith
# 你能看到:
#   - 每个节点的输入输出
#   - 状态在每一步的快照
#   - LLM 请求与响应
#   - 工具调用详情
#   - 执行耗时与 Token 消耗
```

## 与传统编程的类比

| 传统编程 | LangGraph |
|----------|-----------|
| 函数（Function） | 节点（Node） |
| 调用语句 | 边（Edge） |
| if/else | 条件边（Conditional Edge） |
| for/while | 循环边 + recursion_limit |
| 全局变量 | State |
| 参数传递 | State 字段 + Reducer |
| 函数组合 | 子图组合 |
| 断点调试 | interrupt + LangSmith |

## 何时用 LangGraph

### ✅ 适合用 LangGraph

- 需要循环（Agent 工具调用循环）
- 需要条件分支（不同输入走不同路径）
- 需要状态管理（多轮对话记忆）
- 需要人工介入（审批、修正）
- 需要多 Agent 协作
- 需要持久化和恢复

### ❌ 不适合用 LangGraph

- 简单的单次 LLM 调用（直接用 OpenAI SDK）
- 固定的线性流水线（用 LangChain Chain 即可）
- 简单的 Prompt 模板（用 f-string 即可）

## 学习建议

1. **先跑通**：不要一开始就纠结架构，先让第一个程序跑起来
2. **再理解状态**：State 是核心，理解了 State 就理解了 80%
3. **然后掌握条件边**：这是从「固定流程」到「动态流程」的关键
4. **最后学高级特性**：Checkpoint、并行、子图、多 Agent

## 小结

| 核心概念 | 一句话总结 |
|----------|-----------|
| StateGraph | 有状态的图执行引擎 |
| State | 节点间共享的数据容器 |
| Node | 执行计算的函数 |
| Edge | 定义执行顺序 |
| Conditional Edge | 根据条件动态路由 |
| Reducer | 定义状态如何合并 |
| Checkpoint | 状态持久化与恢复 |
| Pregel | 超步（Superstep）执行模型 |
